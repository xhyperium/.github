# 自验证机制规则（Self-Verification）

> 来源：agent-discipline.md（验证是隐式的）+ agent-safety.md（证据优先）+ autonomous-iteration.md（Verify/Guard）+ agent-teams-constitution.md（C-4 证据优先）
> 适用：所有 AI Agent 会话（Claude Code / Codex / Gemini），跨语言、跨项目
> 执行力度：🟢 强制（与 Harness 铁律同级）
> SSOT：`xhyperium/.github` → `rulesets/self-verification.md`

---

## 定位

本规则填补「声称完成」与「确认完成」之间的空白。现有规则已要求「证据优先」（C-4）、「验证是隐式的」（agent-discipline §3），但未定义**完成声明本身必须经过的固定动作序列**。本文件把这个序列显式化为三道关卡：**自查清单 → 结果比对 → 双重确认**，三道关卡缺一不可，顺序不可颠倒。

```
产出 → [关卡一：自查清单] → [关卡二：结果比对] → [关卡三：独立复验] → 声明完成
```

任一关卡不通过，禁止进入下一关卡，禁止声明完成。

---

## 关卡一：完成前自查清单（P0）

**规则**：在输出「完成 / done / 已实现 / 已修复」等结论前，必须在当前上下文中逐项确认以下清单，且每项都要有可指认的依据（文件路径、命令输出、行号），不能仅凭印象打勾。

| 检查项 | 判定标准 | 未通过时的动作 |
| --- | --- | --- |
| **需求覆盖** | 用户原始请求中的每一条诉求都能在改动中找到对应 | 补齐缺口，不得裁剪需求范围后声称完成 |
| **构建/编译** | 项目存在构建步骤时，已执行并通过 | 修复构建错误后重新执行 |
| **测试** | 受影响范围的测试已执行并通过；新增行为有对应测试 | 补测试或修复失败用例 |
| **Lint/格式** | 项目质量门禁（fmt/clippy/eslint 等）已执行并通过 | 修复后重新执行 |
| **无遗留标记** | 代码中无遗留的 `TODO`/`FIXME`/`unimplemented!`/占位符属于本次改动范围 | 补全或转为 follow-up 任务并明确记录 |
| **Scope 边界** | 改动文件集合与任务范围一致，无越界修改 | 回退越界改动或升级范围（需征得同意） |
| **无副作用回归** | 相邻/依赖功能未被本次改动破坏（如有全量测试，已跑过） | 定位回归并修复 |

**禁止**：把这份清单当作 TODO 列表写进任务计划里逐项「打勾了事」——清单检查的证据必须来自当前会话的实际执行（对应 C-4 证据优先），不能引用历史输出或口头声称。

## 关卡二：输出结果与预期结果比对（P0）

**规则**：产出后，必须重新读取原始需求/计划/验收标准（spec、plan、issue 描述、用户原话），逐条对照实际产出，而不是凭记忆判断「应该没问题」。

### 执行方式

1. **重新读取，不用记忆**：重新读取需求来源文件（spec.md / plan.md / issue / 用户消息），禁止依赖对话早期的记忆摘要做比对——记忆会在长对话中漂移（对应 agent-context §1 注意力刷新）。
2. **逐条列出差异**：对每一条验收标准，明确回答「满足 / 不满足 / 部分满足」，不允许模糊的「大致满足」。
3. **量化优先于主观**：凡是能用命令输出、数字、diff 验证的结论，禁止用「应该没问题」「看起来可以」代替（对应 autonomous-iteration §3 指标必须可机械验证）。
4. **不一致时不得静默调整预期**：如果发现实际产出与预期不符，禁止反向修改验收标准去「凑合格」，必须修复产出或明确升级给用户裁决。

### 比对结果分类

| 分类 | 定义 | 后续动作 |
| --- | --- | --- |
| **完全一致** | 每条需求都有对应产出且验证通过 | 进入关卡三 |
| **部分一致** | 核心需求满足，边缘场景/优化项未覆盖 | 明确列出未覆盖项，禁止笼统声称「完成」 |
| **不一致** | 核心需求未满足或产出方向错误 | 停止声明完成，重新实现或升级澄清 |

## 关卡三：双重确认协议（先说后验，P0）

**规则**：这是最容易被跳过的一步——模型倾向于「验证过就等于验证完成了」，把关卡一、二的检查过程本身当成确认。双重确认要求**两个独立的动作**，第二个动作不能是第一个动作的复述。

### 强制顺序

1. **第一步 — 陈述完成理由**：先用一句话明确写出「我认为已完成，依据是 X」，X 必须具体到命令、文件、行号（例如：`` `cargo test --workspace` 全部通过，见上方输出；用户要求的 3 个函数改动均在 `src/foo.rs:12-45` ``）。
2. **第二步 — 独立复验命令**：在陈述之后，**单独执行一条新的验证命令**去确认 X 为真，而不是重新粘贴第一步已经跑过的同一份输出。复验命令必须是能推翻结论的检验，而非确认性检验。

   | 场景 | 第一步（陈述依据） | 第二步（独立复验，不可省略） |
   | --- | --- | --- |
   | 「测试通过了」 | 引用测试输出 | 重新单独跑一次相关测试（不是滚动查看同一次输出） |
   | 「文件已修改」 | 引用 diff | `grep`/`Read` 确认修改内容仍在磁盘上、未被后续操作覆盖 |
   | 「bug 已修复」 | 描述修复逻辑 | 用触发 bug 的原始输入重新复现一次，确认不再触发 |
   | 「配置已生效」 | 引用配置文件内容 | 实际读取运行时状态（进程/服务/命令输出），而非只读配置源文件 |
   | 「依赖已安装/构建已通过」 | 引用安装日志 | 重新执行一次干净的构建/运行命令，而非复用缓存的历史输出 |

3. **两步不可合并**：禁止把「我认为完成，因为刚跑过测试」直接当作双重确认——这是同一个动作，只是换了措辞。第二步必须是**这一轮新执行**的动作。

### 反规避对照（借用 agent-safety §5）

| 规避说法 | 为什么不构成双重确认 |
| --- | --- |
| 「我刚才已经验证过了」 | 关卡一/二的验证是产出阶段的检查，不是完成声明阶段的独立复验 |
| 「逻辑上肯定没问题」 | 逻辑推导不是证据，必须有可执行命令的输出 |
| 「这个改动很小，不需要再验一次」 | 改动大小与是否需要复验无关；关卡三对所有声明完成的场合生效 |
| 「之前类似的改动验证过，这次应该一样」 | 上下文不同，历史验证不能替代当前会话的复验 |

## 决策逻辑

```
IF 关卡一全部通过 AND 关卡二判定为「完全一致」 AND 关卡三双重确认均成立:
    声明完成，附带三道关卡的证据摘要

ELIF 关卡二判定为「部分一致」:
    声明「部分完成」，逐条列出未覆盖项，不使用「完成」措辞

ELIF 任一关卡未通过:
    禁止声明完成
    返回对应关卡定位问题（修复 / 补测试 / 升级澄清）
    修复后必须重新走完三道关卡，不能只补跑失败的那一项
```

## 适用范围与豁免

| 场景 | 是否豁免 | 说明 |
| --- | --- | --- |
| 探索性/只读调研任务 | ✅ 豁免关卡二、三 | 无产出即无需比对预期与结果 |
| 单文件 typo/注释修复 | ✅ 可简化关卡一为「读一次确认改动」 | 仍需关卡三的最小复验（重新 `Read` 确认文件内容） |
| 紧急生产热修复 | ⚠️ 关卡一可延后 | 修复后 72 小时内补齐完整三道关卡，对应 agent-teams-constitution.md §9 豁免规则 |
| 常规功能开发/修复/重构 | ❌ 不豁免 | 三道关卡全部强制 |
